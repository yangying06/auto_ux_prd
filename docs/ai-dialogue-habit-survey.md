## 一、整体结论

你的 AI 使用风格非常明显：**目标导向、迭代频繁、敢于把真实工程问题直接丢给 AI、并且愿意持续追打直到结果正确**。

这是一种很强的使用方式，优点是：
- 非常适合推进具体开发任务；
- 非常适合 bug 定位、表现调优、文档补齐、工具链排障；
- 能把 AI 当成“持续协作的实现伙伴”，而不是只拿来问概念。

但从历史看，也有几个会持续拉低效率的问题：
- **需求常常先给大方向，细约束后补**，导致 AI 第一版容易做偏；
- **纠错时常直接说“还是不对”“有问题”**，但没有第一时间给出“偏差坐标系”；
- **经常跨会话/跨工作流切换**，有时会造成上下文断裂；
- **会重复尝试命令、模型切换、继续/exit 等操作**，说明工具流还没有完全稳定下来；
- **你对结果正确性要求很高，但对验收标准经常是边看边补**，这会增加来回轮次。

一句话总结：

> 你已经具备了“把 AI 当工程搭子”的成熟习惯，但如果再把**任务定义、验收标准、纠错反馈**三件事前置得更明确，整体效率会明显再上一个台阶。

## 四、你目前最影响效率的几个问题

## 1. 需求 often 是“先大后细”，导致第一版容易跑偏

你的很多提问，会先给一个目标，但关键约束会在后续几轮才补出来。

例如常见模式是：
1. 先说“实现某个功能”；
2. AI 做出第一版；
3. 你再补“不是这样，应该按这个时序”；
4. 再补“这里要静止，那里要循环”；
5. 再补“多个目标时顺序要从左到右、从上到下”。

这会导致 AI 第一轮方案天然带有猜测成分。

### 改进建议

以后如果是功能实现类请求，建议直接按这个模板发：

```md
目标：
- 我要实现什么

输入：
- 数据从哪里来
- 关键字段是什么

表现规则：
1. 第一步发生什么
2. 第二步发生什么
3. 第三步发生什么

禁止事项：
- 不能怎样
- 不要改哪些现有逻辑

验收标准：
- 我看到什么现象说明对了
- 我看到什么现象说明错了
```

这会极大减少“先做错再纠偏”的轮次。

## 2. 纠错时有时太短，只说“还是不对”

你历史里有不少这种反馈：
- “还是不行”
- “依然没有修改正确”
- “还是这样”
- “有问题”

这种反馈当然是真实的，但对 AI 来说信息密度偏低。AI 不知道到底是：
- 逻辑错了；
- 时序错了；
- 坐标错了；
- 样式错了；
- 只错了 10 像素；
- 还是整套方向错了。

### 更高效的纠错写法

建议你把纠错固定成下面这种格式：

```md
还是不对，具体偏差是：
1. 预期：xxx
   实际：xxx
2. 预期：xxx
   实际：xxx
3. 当前可以先不要动 xxx，只修 xxx
```

如果是 UI 问题，再多补一句：
- “看起来像是中心点没对齐”
- “像是多加了半个 label 高度”
- “像是停止时机错了，不是位置错了”

你其实已经偶尔会这样做，比如：
- “看表现是高度高了 h/2”
- “需要在盘面开始转动时就移动火圈”

这类反馈质量就很高，建议保持并系统化。

## 3. 会话切换、exit、继续、重复命令比较多

历史里能看到不少：
- `exit`
- `继续`
- `/new`
- 重复 `/model`
- 重复 `hi`
- 同一条请求短时间重复发送

这通常意味着几种情况：
- 当前会话上下文不稳定；
- 不确定当前命令是否生效；
- 想换上下文但没有固定切换习惯；
- 工具链本身存在状态问题。

### 影响

这会导致：
- 任务上下文断裂；
- AI 丢失前面约束；
- 你自己也要重新描述一部分内容；
- 重复消耗轮次。

### 改进建议

尽量给每个会话一个单一目标，例如：
- 会话 A：只做一个功能实现；
- 会话 B：只查一个 bug；
- 会话 C：只做 push / 提交 / 工作流；
- 会话 D：只做文档整理。

并且开头固定一句：

```md
这个会话只做一件事：xxx。
不要切到别的话题，直到它完成。
```

会明显稳定很多。

## 4. 你经常在“实现中”才补验收标准

你对结果要求高，但很多验收标准是看到结果后才补出来的，例如：
- “label 需要立即出现，而不是等结算时”
- “需要居中”
- “需要使用 number.fnt”
- “被火圈套中时才播放旋转动画”
- “移出盘面后等停止才消失”

这些其实都属于很关键的验收条件。如果前置给出，AI 的首版命中率会高很多。

### 建议

实现类任务尽量在第一条消息里补一句：

```md
验收标准：
- A 对
- B 对
- C 对
- 如果出现 D 就说明做错了
```

这是你当前最值得优化的一点。

## 5. 偶尔会让 AI 做“高风险动作”，但边界描述不够完整

例如：
- 帮我 push
- 如果遇到冲突，以远端为准
- 继续帮我 push 上去
- 检查本地是否还有内容没有提交

这些都是真实需求，但它们属于**高风险工程动作**。如果边界不清楚，AI 很容易：
- 覆盖本地改动；
- 错误解决冲突；
- 提交不该提交的内容；
- 把未验证代码推上去。

后面你也确实遇到过：
- 更新后一直显示“游戏暂停中”
- 怀疑冲突解决方式不对

### 更安全的提法

以后建议写成：

```md
请先只做下面几件事：
1. 检查哪些文件未提交
2. 告诉我将要提交什么
3. 如果有冲突，不要自动处理，先汇报冲突文件和两边差异
4. 只有我确认后再 push
```

这样更稳。

## 七、具体可执行的改进建议

## 1. 把“验收标准”前置

这是收益最大的优化项。

你现在的问题不是不会提需求，而是很多“判断对错的标准”出现在第二轮、第三轮才说。

建议：每次实现类请求都强制自己补 3 条验收标准。

## 2. 纠错时尽量用“预期/实际”对照

不要只说“还是不对”。

改成：
- 预期：xxx
- 实际：xxx

这样 AI 会收敛得快很多。

## 3. 每个会话只处理一个主任务

少一点：
- 来回 exit
- 继续
- 再开新会话补前文

多一点：
- 这个会话只做 XXX
- 完成前不切话题

## 4. 对 AI 明确“先分析还是先改”

你有时其实想先要判断，但提法像是直接让 AI 动手。

例如可以明确写：
- “先分析原因，不要改代码”
- “先列实现方案，再执行”
- “直接修改，不用先给方案”

这会让协作方式更稳。

## 5. push / pull / merge 类操作一定加保护栏

尤其是：
- 冲突处理
- 远端优先
- 覆盖本地
- 继续提交

建议都先让 AI 做只读检查，再由你确认。

## 6. 把常用提示词沉淀成你自己的固定模板

你已经有明显的高频场景了，完全值得固化成模板：
- 功能实现模板
- bug 排查模板
- UI 调整模板
- 提交前检查模板
- 文档整理模板

一旦模板化，你的 AI 使用效率会再提升一截。

1. 只会问概念

只要把这三点再收紧，你和 AI 的协作质量会明显更稳定。
